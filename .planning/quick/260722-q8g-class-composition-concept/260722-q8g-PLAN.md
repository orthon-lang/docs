<objective>
Create a new concept research document, `how/concepts/research/CLASS_OR_STRUCTURE_AS_PRIMARY_COMPOSITION.md`,
documenting a recommended composition pattern synthesized from Python idioms: a data-only
aggregate type (dataclass-equivalent) as the primary composition unit, layered with stateless
behavior mixins (no fields, no cooperative `super()` chains) for cross-cutting magic-method-style
behavior, and structural/protocol-style interfaces used purely for static contract-checking
without runtime inheritance. The document frames these as three orthogonal tools that are
combined per-problem — not a mandatory trio — and translates the pattern (plus its caveats)
into implications for Orthon's existing derive/mixin/trait research.

Purpose: The user's own notes (originally in Russian) argue this three-tool combination is the
"optimal gentleman's set" for Python-style composition: dataclass power (auto `__init__`/`__repr__`/`__eq__`),
composition safety (no deep hierarchies), duck-typing flexibility, and static interface checking
(Go-like) — but the three tools are not always all needed together. This repository already has
separate research on each ingredient (DATACLASSES.md, MIXIN.md, STRUCTURAL_TYPING.md) and on the
general composition-vs-inheritance philosophy (COMPOSITION_OVER_INHERITANCE.md), but nothing yet
captures the specific "when do you reach for one, two, or all three" recipe, nor the caveats for
when this pattern is overkill. This document closes that gap as a DRAFT research artifact — it
does not itself decide Orthon's final trait/derive semantics (that happens at EDR acceptance).

Output: `how/concepts/research/CLASS_OR_STRUCTURE_AS_PRIMARY_COMPOSITION.md`, a DRAFT concept
document following the `_concept.md` template exactly, cross-referencing rather than duplicating
DATACLASSES.md, MIXIN.md, COMPOSITION_OVER_INHERITANCE.md, and STRUCTURAL_TYPING.md.

    tools should a programmer reach for to compose a typical type, and when is using all of them
    overkill? State the problem concretely: Python offers three independent, non-overlapping
    tools — `@dataclass` (data + boilerplate reduction, see DATACLASSES.md), mixin classes with no

    realizing they compose orthogonally. State the core problem the document answers: should
    Orthon's default type/trait model bless a specific "gentleman's set" combination of these three
    orthogonal tools as an idiomatic composition pattern, and how does the language make clear when
    only a subset is needed?

    **`## Principles`** — List these five, each with a one-sentence tie to the pattern and a
    citation:
    1. Orthogonality (`../DESIGN_PRINCIPLES.md` § Orthogonality) — each tool solves exactly one
       problem (data layout, behavior reuse, contract checking) and none should be bundled into a
       single mandatory mechanism.
    2. Composition over inheritance (`COMPOSITION_OVER_INHERITANCE.md` Principle 1) — none of the
       three tools should require or encourage deep hierarchy.
    3. Single Responsibility (`../DESIGN_PRINCIPLES.md` § SOLID) — a data aggregate's only
       responsibility is holding data; a behavior mixin's only responsibility is one cross-cutting
       concern; an interface's only responsibility is declaring a contract.
    4. Explicitness (`../DESIGN_PRINCIPLES.md` § Explicitness, and `MIXIN.md` Principle 2) — mixins
       must not rely on cooperative `super()`/MRO traversal; conflicts must be visible at the
       composition site.
    5. Minimal Core (`../DESIGN_PRINCIPLES.md` § Minimal Core) — this pattern must not require a
       new language construct; it should be achievable purely by composing Orthon's existing
       derive mechanism (DATACLASSES.md), trait/mixin mechanism (MIXIN.md, TRAITS.md), and
       structural trait satisfaction (STRUCTURAL_TYPING.md).

    cooperative `super()` chains, per MIXIN.md's rejection of implicit linearization) is added only
    when the *same* cross-cutting magic-method-style logic repeats across two or more otherwise
    unrelated data aggregates; (3) a structural/protocol-style interface (per STRUCTURAL_TYPING.md's
    structural trait satisfaction) is added only when a caller needs a statically-checkable contract
    without forcing the data aggregate to declare or inherit anything.

    never declares `Addable` anywhere — the `Protocol` is satisfied structurally and checked only
    by a static type checker, while `AddMixin` supplies the one cross-cutting behavior and the
    `@dataclass` supplies everything data-shaped, with each of the three tools doing exactly one
    job and none of them aware of the others.

    the data aggregate maps to a `derive`-annotated type (DATACLASSES.md); the interface/contract
    maps to a trait declaration used purely for its signatures, satisfied per whichever
    satisfaction policy Orthon settles on; the cross-cutting mixin maps to a trait carrying default
    method implementations (TRAITS.md's "Default Implementations" subsection, and MIXIN.md's

    conflict between TRAITS.md and MIXIN.md/STRUCTURAL_TYPING.md is unresolved and must be settled

    unconditionally rather than by an optional external tool, one of the Python "when NOT to use
    this" caveats (see Alternative Strategies below) does not carry over to Orthon by construction.

    **`## Default Strategy`** — State that by default a composed type in Orthon is modeled as a
    `derive`-annotated data aggregate for state, implementing zero or more traits for contracts;
    behavior-carrying (mixin-role) traits are opt-in and added only once duplicate trait-default
    logic is identified across two or more otherwise-unrelated aggregate types, never applied
    speculatively to a single type.

    **`## Alternative Strategies`** — List, as permitted reduced combinations: (1) data aggregate
    alone, no trait and no mixin — appropriate for simple config/DTO/value types with no behavior
    beyond structural equality/repr; (2) data aggregate plus one or more traits, no mixin — when
    each trait's implementation is unique to that one type, so a default-impl-bearing trait would
    be unnecessary abstraction; (3) data aggregate plus a mixin-role trait, no separate
    contract-only trait — when the shared behavior never needs to be a generic constraint
    elsewhere; (4) the full three-tool composition — when a cross-cutting behavior and a
    checkable generic constraint are both needed. Then list, translated into Orthon terms, the
    three "when NOT to use this pattern at all" caveats from the source notes: single, simple,

    tooling (in Python this caveat exists because `typing.Protocol` is unenforced without a
    separate type checker like mypy — cross-reference back to the "Translating to Orthon" note
