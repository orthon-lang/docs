# LLM Generability Gate

**Status:** *Accepted — see EDR-011 for the formal decision.*

**Date:** 2026-07-19

**Context:** Discussion about whether Orthon (as an LLM-native language)
needs a dedicated validation gate for LLM-specific concerns, and whether
the ROADMAP requires adaptation.

## Problem

Orthon defines itself as an LLM-native language (VISION.md pillar).
However, the current decision validation system (6 gates in
`DECISION_VALIDATION.md`) has no gate that validates whether a
language construct is LLM-generable. Existing gates check for:

- User value (`USER_VALUE_GATE`)
- Logical consistency (`LOGICAL_CONSISTENCY_GATE`)
- Conceptual simplicity (`CONCEPTUAL_SIMPLICITY_GATE`)
- Architectural fit (`ARCHITECTURAL_INTEGRITY_GATE`)
- Strategy independence (`IMPLEMENTATION_INDEPENDENCE_GATE`)
- Long-term maintainability (`LONG_TERM_MAINTAINABILITY_GATE`)

None of these asks: *"Can an LLM generate correct code using this
construct with acceptable probability?"* — which is a core requirement
for an LLM-native language.

Additionally, the ROADMAP doesn't explicitly account for LLM Toolchain
components or LLM-specific validation in any milestone.

---

## Proposal: LLM_GENERABILITY_GATE

Add a 7th gate to the Decision Validation system.

### Core Question

*Is the concept LLM-generable — can an LLM reliably produce correct
code using this construct?*

### Method

Empirical analysis with two complementary approaches:

1. **Structural analysis** — examine the concept's grammar and semantics
   for known LLM-unfriendly patterns (ambiguity, special cases,
   context-dependent syntax, tokenizer misalignment).

2. **Schema round-trip** — verify that the concept can be fully
   expressed in a machine-readable schema (grammar + type system +
   stdlib contracts), and that an LLM consuming this schema would
   have enough information to generate correct code.

### Criteria

| Criterion | Pass | Flag | Fail |
|-----------|------|------|------|
| **Schema-serializable** — the concept is fully expressible in a machine-readable grammar/type schema | Full representation in schema without loss | Partial representation (minor loss of constraints) | Cannot be represented in schema |
| **Predictable generation** — LLM generates the construct correctly in ≥90% of attempts | No known ambiguity or special cases | One identified edge case where LLMs systematically err | Multiple failure modes identified |
| **No hallucination surface** — no ambiguous or confusable constructs | Every form has exactly one meaning | One ambiguous parse possible | Multiple ambiguous or confusable forms |
| **Strategy-aware default** — code generated by default is valid under any Implementation Strategy | Always valid without annotations | Valid under most strategies; requires explicit Policy annotation for some | Generated code systematically violates Strategy constraints |
| **Self-correctable** — Static Analyser can detect and report LLM-generation errors | Full diagnostic coverage (type errors, policy violations, strategy mismatches) | Partial coverage with silent gaps | No or minimal diagnostic support |

### Decision Type Mapping

| Decision Type | Gates Typically Required | Notes |
|---------------|-------------------------|-------|
| New language construct | All **7** gates | LLM_GENERABILITY added to all-six baseline |
| LLM Toolchain component | `ARCHITECTURAL_INTEGRITY_GATE`, `LLM_GENERABILITY_GATE`, `USER_VALUE_GATE` | Toolchain components affect LLM-native experience directly |
| Syntax change | `LOGICAL_CONSISTENCY_GATE`, `CONCEPTUAL_SIMPLICITY_GATE`, `ARCHITECTURAL_INTEGRITY_GATE`, `LLM_GENERABILITY_GATE` | Syntax changes affect LLM generation most directly |

### `_language-design.md` Checklist Addition

Add the following criterion to the language design gate checklist:

```
☐ LLM generability — can an LLM generate correct code using
  this construct?
  • The concept is expressible in a machine-readable schema (grammar +
    type system + stdlib contracts)
  • No ambiguities that an LLM could misinterpret
  • The Static Analyser can detect LLM-generation errors for
    this construct
  • Code generated by default is correct for all Implementation
    Strategies (or explicitly requires a Policy annotation)
```

## ROADMAP Adaptation

### Milestone 1 (Language Inventory)

Add to deliverables:
- LLM Toolchain components in `LANGUAGE_INVENTORY.md`:
  Schema Provider, Code Completer, Code Generator, Static Analyser,
  Documentation Generator, Refactor/Migration Tool
- LLM-related Policy Types: Toolchain Policy, Error Policy,
  Inference Policy

### Milestone 2 (Concept Design Review)

- **New note:** LLM Toolchain components pass Concept Design Review
  as a **parallel track**, using gates: `ARCHITECTURAL_INTEGRITY_GATE`,
  `USER_VALUE_GATE`, `LLM_GENERABILITY_GATE`.
- Each language construct, when reviewed, must pass all 7 gates
  (the new `LLM_GENERABILITY_GATE` included).

### Milestone 5 (Language Specification)

Add deliverable:
- `SCHEMA.md` — machine-readable language schema (grammar, types,
  stdlib contracts) in a format consumable by LLM Toolchain

### Milestone 6 (Validation)

Add to deliverables:
- **LLM generation tests** — validate that LLMs can generate
  correct Orthon code for each concept
- **Schema round-trip tests** — verify schema → generation →
  validation cycle end-to-end

### Milestone 8 (Standard Library & FFI)

Add subsection:
```
| 8.3 | LLM Toolchain | Schema Provider, Code Completer, Code Generator, Static Analyser implementation |
```

## Open Questions

1. **Method refinement** — Is "empirical analysis" sufficient, or
   should the gate prescribe a specific method (like Socratic for
   LOGICAL_CONSISTENCY, TRIZ for IMPLEMENTATION_INDEPENDENCE)?
   Possible dedicated method: confusability analysis + schema
   round-trip testing.

2. **Pass threshold** — 90% correct generation is proposed as Pass
   threshold. Should this be formalised through empirical testing
   during Milestone 6 instead?

3. **LLM Toolchain concepts vs Language concepts** — Should LLM
   Toolchain components (Schema Provider, etc.) use the same Concept
   Design Review process as language constructs, or a lighter process?
   Current proposal: lighter (3 gates instead of 7).
