## Issue (Why)

How does a programmer construct a complex object with many optional parameters without writing a Builder class?

In Java, the Builder pattern is the standard solution for objects with many optional fields:

```java
Pizza pizza = new Pizza.Builder()
    .size(12)
    .cheese(true)
    .pepperoni(true)
    .extraToppings(List.of("olives", "mushrooms"))
    .build();
```

The Builder pattern requires:
- A separate `Builder` class with all the same fields.
- A fluent setter method for each optional field.
- A `build()` method that constructs the target object.

This is ~30-50 lines of boilerplate per type — code that is identical in structure across every Builder.

The core problem: **can language features (named arguments, default parameters, immutable construction, DSLs) eliminate the Builder pattern entirely, so the above becomes:**

```
pizza = Pizza(size: 12, cheese: true, pepperoni: true, toppings: ["olives", "mushrooms"])
```

Or even more concisely:

```
pizza = Pizza {
    size = 12
    cheese = true
    pepperoni = true
    toppings = ["olives", "mushrooms"]
}
```

## Examples

| Language | Builder needed? | Alternative |
|---|---|---|
| **Java** | Yes — no default args, no named params | Builder pattern is idiomatic |
| **Kotlin** | Rarely | Named args + default params + `apply` DSL |
| **Scala** | Rarely | Named args + default params + type-safe builders |
| **Python** | Rarely | `**kwargs` + dataclass defaults |
| **Rust** | Sometimes | Builder pattern via builder crate, or `..default()` |
| **Swift** | Rarely | Named args + default values on `init` |

## Implications for Orthon

1. **Named arguments make Builder unnecessary for most cases** — Every constructor/function call can specify arguments by name. Combined with default values, the caller provides only the non-default fields:

   ```
   struct Pizza:
       size: Int = 10
       cheese: Bool = false
       pepperoni: Bool = false
       toppings: List[String] = []

   // Usage — only specify non-defaults
   pizza1 = Pizza()
   pizza2 = Pizza(size: 12, pepperoni: true)
   pizza3 = Pizza(size: 14, cheese: true, toppings: ["olives"])
   ```

2. **Copy with modifications** — For immutable objects, a `copy` method (or syntax) creates a new instance with specific fields changed:

   ```
   pizza4 = pizza3.copy(toppings: ["mushrooms"])
   ```

3. **Builder DSL via lambdas** — For deeply nested or imperative-style construction, a lambda-based DSL can be provided by the type author, not generated by boilerplate:

   ```
   pizza = build Pizza:
       size = 12
       cheese = true
       toppings.add("olives")
   ```

   This is library-level, not compiler-generated.

4. **No separate Builder class** — Because named args + defaults cover the same use case, the language never requires a separate Builder type. The programmer writes one declaration (`struct`), not two.

5. **When Builder is still needed** — The Builder pattern remains useful for:
   - Complex validation across multiple fields (cross-field invariants).
   - Step-by-step construction where intermediate state is meaningful.
   - Fluent API design (e.g., query builders, test data builders).
   
   In these cases, a Builder is intentional, not a workaround for language limitations.

## Open Questions

1. Should `copy` be auto-generated for all structural types (like Kotlin's `data class`), or opt-in?
2. Should the language provide a built-in `build` function for DSL-style construction, or leave this to libraries?
3. Does the copy-with-modification pattern require a dedicated syntax (`obj with { field = new_value }`) or is a `copy` method sufficient?
4. How does Builder elimination interact with validation — if a `struct` has invariants (e.g., `size > 0`), can they be checked at construction time without a Builder?
