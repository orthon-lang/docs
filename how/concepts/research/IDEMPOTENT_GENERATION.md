## Issue (Why)

When an LLM generates the same code fragment twice — the same function, the same module, the same expression — should the result be semantically equivalent?

Current reality:

- **LLM temperature and sampling** — The same prompt produces different code on every generation. Sometimes these differences are cosmetic (different variable names, different comment style). Sometimes they are semantic (different algorithm, different error handling, different side effects).
- **Hidden global state** — Many languages have constructs whose meaning depends on implicit global state: module-level mutable state, process-wide singletons, ambient authority (system clock, random seed, environment variables, filesystem). Two generations of the same function may produce different results because the surrounding state changed.
- **Import order sensitivity** — In Python and JavaScript, the order of imports can change runtime behaviour. An LLM generating a module has no control over what was imported before it.
- **Non-deterministic defaults** — `HashMap` iteration order, `random` seed, `Date()` — in many languages, the same code run twice produces different results. An LLM cannot guarantee that its generated code is deterministic without explicit annotations.

The core problem: **if the same prompt generates semantically different code on two invocations, the LLM cannot self-verify its own output, and the user cannot trust that a regenerated solution is equivalent to the previous one**.

For Orthon, this means:

1. **LLM self-consistency** — The language and standard library must be designed so that the same specification (expressed as a prompt or schema) leads to the same semantic result, modulo cosmetic variation.

2. **Predictable composition** — If module A was generated correctly and module B was generated correctly, their composition (A + B) should also be correct, independent of the order in which they were generated.

3. **Reproducible builds** — A program generated by an LLM should compile to the same binary regardless of when, where, or in what order the generation occurred.

## Principles

1. **No hidden global state affecting semantics** — The language has no implicit global state that changes program meaning. Module-level state is explicit and visible. Global singletons are prohibited unless explicitly declared with `global`.

2. **Deterministic iteration order** — All built-in collections iterate in a deterministic, documented order. Hash-based collections use a stable hash (key order, not insertion or hash order). Non-deterministic iteration must be explicitly requested.

3. **Explicit time and randomness** — `Date()`, `Random()`, `UUID()`, and similar time- or entropy-dependent functions return values based on explicitly passed context, not ambient state. A `Context` parameter must be threaded through or a generator must be explicitly created.

4. **Import-order independence** — Module initialisation order does not affect runtime semantics. Circular imports are compile-time errors. Module-level side effects are prohibited (module initialisation propagates constants and types, not mutable state).

5. **Canonical representation** — The same source code (after formatting) always produces the same compiled output. No build-timestamp, hostname, or path-dependent output in debug symbols that affect behaviour (debug metadata is separate).

6. **Semantic equivalence under regeneration** — Two LLM generations of the same specification produce code that is semantically equivalent: same preconditions, postconditions, and observable behaviour, even if variable names and comments differ.

## Model (What)

### Deterministic Collections

All standard collections iterate in a deterministic order:

```orthon
let map = Map<String, Int>.new()
map.insert("c", 3)
map.insert("a", 1)
map.insert("b", 2)

for (key, value) in map:
    # Always produces: ("c", 3), ("a", 1), ("b", 2)
    # — insertion order, not hash order
    print(key, value)
```

For non-deterministic iteration, an explicit opt-in is required:

```orthon
let map = Map<String, Int>.new(order: Unordered)
# Iteration order is unspecified
```

### Explicit Context for Time and Randomness

Time-dependent operations receive their time source explicitly:

```orthon
fn log_timestamp(ctx: Context)
    let now = ctx.clock.now()    # explicit: which clock?
    print("[{now}] event logged")
```

Or via a default parameter that can be overridden:

```orthon
fn now(clock: Clock = Clock.system) -> DateTime
    clock.now()

# In tests:
let mock_clock = Clock.fixed(DateTime(2026, 1, 1))
let t = now(clock: mock_clock)   # Always returns 2026-01-01
```

Similarly for randomness:

```orthon
let rng = Rng.seed(42)           # Deterministic
let value = rng.next_float()     # Always the same for seed 42
```

### Import-Order Independence

Module initialisation has strict ordering guarantees:

```orthon
module math_utils
# Module-level code may only:
# - Define constants (compile-time evaluable)
# - Define types and functions
# - Re-export symbols from dependencies
# It must NOT:
# - Execute I/O
# - Mutate global state
# - Call functions with side effects
```

```orthon
module service
# This is a compile-time error if module-level code has side effects:
let global_state = MutableState.new()
# Error: Module-level initialisation must be pure

# Correct:
let CONFIG = Config.load("app.toml")
# Error: File I/O is a side effect at module level

# Correct approach:
let config = Config.load("app.toml")  # inside a function
```

### Semantic Equivalence Under Regeneration

The compiler provides a `--check-equivalence` mode that compares two compiled modules for semantic equivalence (same public API, same contract effects, same type structure):

```bash
wvy check-equivalence old_version.wvy new_version.wvy
# Output: Semantically equivalent (cosmetic differences only)
#   or: Semantic difference detected: function `sort` has different
#       precondition in new version
```

```
                - Deterministic collections (enforced by stdlib design)
                - Explicit RNG and Clock (stdlib API design)
                - Semantic equivalence checker (toolchain tool)

                However, some aspects require language-level changes:
                - Prohibition of module-level side effects
                - Deterministic iteration guarantee on built-in types
                - Import-order independence (language specification)

                → MIXED — most aspects are Level 3 (Stdlib) but some
                  require language-level specification.

Pattern test:   Can Idempotent Generation be expressed as a composition
                of existing Primitive Operations and Data Model constructs?

                The principle does not introduce new primitives. It
                constrains how existing primitives are used:
                - Module initialisation using only `assignment` + `literal`
                - Collections built from `pack` + deterministic iteration
                - Time/randomness as first-class values (`function` + `call`)
                These are all compositions of existing Level 0–1 constructs.

                → PASS — belongs in Language Patterns (Level 2).

Filter result: Level 2 (Language Pattern), with significant Level 3
               (Standard Library) aspects for collections, RNG, and
               time APIs.
```

## Default Strategy

The Default Strategy enforces full idempotent generation discipline:
- Module-level side effects are compile-time errors
- All built-in collections iterate deterministically
- Time and randomness require explicit context
- The `--check-equivalence` toolchain flag is available

The LLM Strategy relaxes some restrictions (allowing module-level initialisation with side effects for rapid prototyping) but retains the deterministic collection guarantee.

## Alternative Strategies

| Strategy | Description |
|---|---|
| **Full determinism** | All of the above enforced. Suitable for production code where reproducibility matters. |
| **Relaxed** | Module-level side effects are allowed. Collection iteration is deterministic by default but can be opted out. Time/randomness may be implicit. Suitable for scripting and rapid prototyping. |
| **Free** | No idempotency guarantees. Collections may be non-deterministic, module initialisation may have side effects, time/randomness are implicit. Suitable for throwaway scripts. |
| **Principle-only** | The principle is recorded in `DESIGN_PRINCIPLES.md` but no language or stdlib enforcement exists. Tooling and convention are expected to achieve idempotent generation. |

## Open Questions

1. Should the `--check-equivalence` tool be part of the compiler or a separate toolchain component?
2. Should module-level side effects be a hard error or a warning that becomes an error in production builds?
3. How does the deterministic iteration guarantee interact with concurrent modification?
4. Should the standard library provide `DeterministicRng` and `DeterministicClock` wrappers for testing, or should all RNG/Clock types be inherently deterministic?
5. How does import-order independence interact with lazy initialisation patterns?
6. Should the language provide an `@idempotent` annotation on functions to assert that the function is semantically pure and deterministic?
