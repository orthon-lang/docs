# EDR-011: LLM Generability Gate

**Status:** Accepted

**Date:** 2026-07-19

**Category:** Architecture

**Scope:** Project

---

### Context

Orthon defines itself as an LLM-native language. The Vision document
(`docs/why/VISION.md`) lists "Designed for the LLM Era" as one of three
pillars, alongside "Language Designed Like Software" and "Comfortable by
Design." The Manifesto, Architecture, and LLM Strategy all reflect this
commitment.

However, the existing decision validation system — six gates defined in
`docs/how/gates/DECISION_VALIDATION.md` — has no validation dimension
that asks whether a language construct is reliably generable by an LLM:

| Gate | Checks for | LLM dimension? |
|------|------------|:-:|
| `USER_VALUE_GATE` | Solves a real user problem | ❌ |
| `LOGICAL_CONSISTENCY_GATE` | No contradictions | ❌ |
| `CONCEPTUAL_SIMPLICITY_GATE` | Minimal, composable | ❌ |
| `ARCHITECTURAL_INTEGRITY_GATE` | Fits layered architecture | ❌ |
| `IMPLEMENTATION_INDEPENDENCE_GATE` | Strategy-agnostic semantics | ❌ |
| `LONG_TERM_MAINTAINABILITY_GATE` | Ages well | ❌ |

None verifies schema-serializability, hallucination risk, or
predictability of LLM-generated code for a given construct. For a
language that markets itself as LLM-native, this is a blind spot.

The design tension: existing gates could be extended to cover LLM
concerns, but LLM generability is a *separate dimension* of language
quality — not reducible to minimality (CONCEPTUAL_SIMPLICITY),
consistency (LOGICAL_CONSISTENCY), or any other existing gate. It
merits its own validation lens, analogous to how
`IMPLEMENTATION_INDEPENDENCE_GATE` is a separate dimension for the
strategy-agnosticism requirement.

See `docs/notes/llm-generability-gate.md` for the full analysis.

---

### Decision

Add `LLM_GENERABILITY_GATE` as a **7th gate** in the Decision Validation
system (`docs/how/gates/DECISION_VALIDATION.md`), with the following
definition:

#### Core Question

*Is the concept LLM-generable — can an LLM reliably produce correct
code using this construct?*

#### Method

Empirical analysis with two complementary approaches:

1. **Structural analysis** — examine the concept's grammar and semantics
   for known LLM-unfriendly patterns (ambiguity, special cases,
   context-dependent syntax, tokenizer misalignment).

2. **Schema round-trip** — verify that the concept can be fully
   expressed in a machine-readable schema (grammar + type system +
   stdlib contracts), and that an LLM consuming this schema would
   have enough information to generate correct code.

#### Criteria

| Criterion | Pass | Flag | Fail |
|-----------|------|------|------|
| **Schema-serializable** — concept expressible in machine-readable grammar/type schema | Full representation without loss | Partial loss of constraints | Cannot be represented |
| **Predictable generation** — LLM generates correctly in ≥90% of attempts | No known ambiguity or special cases | One edge case where LLMs err systematically | Multiple failure modes |
| **No hallucination surface** — no ambiguous or confusable constructs | Every form has exactly one meaning | One ambiguous parse possible | Multiple ambiguous forms |
| **Strategy-aware default** — generated code valid under any Implementation Strategy | Always valid without annotations | Valid under most; requires Policy annotation for some | Systematically violates Strategy |
| **Self-correctable** — Static Analyser detects LLM-generation errors | Full diagnostic coverage | Partial coverage | No diagnostic support |

#### Decision Type Mapping

| Decision Type | Gates Required | Notes |
|---------------|----------------|-------|
| New language construct | All **7** gates | Baseline |
| LLM Toolchain component | `ARCHITECTURAL_INTEGRITY`, `LLM_GENERABILITY`, `USER_VALUE` | Lighter process |
| Syntax change | `LOGICAL_CONSISTENCY`, `CONCEPTUAL_SIMPLICITY`, `ARCHITECTURAL_INTEGRITY`, `LLM_GENERABILITY` | Syntax affects generation most |

#### `_language-design.md` Checklist Addition

Add to `docs/how/gates/_language-design.md`:

```
☐ LLM generability — can an LLM generate correct code using
  this construct?
  • The concept is expressible in a machine-readable schema (grammar +
    type system + stdlib contracts)
  • No ambiguities that an LLM could misinterpret
  • The Static Analyser can detect LLM-generation errors for
    this construct
  • Code generated by default is correct for all Implementation
    Strategies (or explicitly requires a Policy annotation)
```

---

### Consequences

- **Positive:**
  - All language constructs are explicitly validated for LLM generability
    alongside existing quality dimensions.
  - The LLM-native pillar of VISION.md now has a concrete, enforceable
    mechanism in the design process.
  - Schema serializability becomes a first-class requirement, which
    also benefits tooling (IDE, documentation generators) beyond LLMs.
  - The gate provides clear criteria for *when* a concept is
    "LLM-ready" — reducing ambiguity in design reviews.

- **Negative:**
  - Adds one more gate to the design review process (6 → 7), increasing
    review overhead for new concepts.
  - The "Predictable generation" criterion is empirical and may be hard
    to evaluate before Milestone 6 (Validation) provides real LLM
    testing data.
  - The 90% pass threshold is a proposal; its practical calibration
    requires experience.

- **Open questions (deferred):**
  - Whether the gate should prescribe a named method (e.g.,
    confusability analysis + schema round-trip testing) or remain
    method-agnostic with "empirical analysis" as the default.
  - Whether LLM Toolchain components should use a lighter process
    (3 gates vs 7) or the full Concept Design Review.

---

### Compliance

- The `_language-design.md` checklist gains a new criterion:
  `☐ LLM generability`.
- For each new language construct in Milestone 2, the gate
  `LLM_GENERABILITY_GATE` must be applied alongside the other 6 gates.
- `DECISION_VALIDATION.md` table "Decision Type" updated to show
  all-7-gates for new language constructs.
- The Decision Journal in `_language-design.md` will record
  LLM_GENERABILITY_GATE results for each concept.

---

### Alternatives Considered

| Alternative | Rationale for Rejection |
|-------------|-------------------------|
| **Extend existing gates** (e.g., add LLM criteria to CONCEPTUAL_SIMPLICITY or LOGICAL_CONSISTENCY) | LLM generability is a distinct dimension not reducible to minimality or consistency. Embedding it elsewhere dilutes both gates. Same reasoning as having a separate `IMPLEMENTATION_INDEPENDENCE_GATE`. |
| **No gate — rely on design principles only** | Principles without enforceable gates are aspirational. The existing 6-gate system shows Orthon's commitment to verifiable design quality; LLM-native deserves the same. |
| **Postpone to Milestone 6 (Validation)** | Validation finds issues but doesn't prevent them. The gate must exist during design (Milestone 2) to influence concept architecture before specification. |
